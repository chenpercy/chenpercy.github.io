Determine if Two Strings Are Close

Posted on 2021-01-25 Edited on 2021-11-23 In Program

出處:

Leetcode Week 4: Jan. 22nd - Jan. 28th

題目:

給定兩字串，並根據以下兩條規則進行操作，則這兩字串視為密切相似的

  • 任意交換兩個 已存在的 字元，也就是說 abcde -> aecdb
  • 轉換任意 已存在的 的字元為另一個 以存在的 字元，也就是說 abbcd -> addcb (注意! 相同的字元都要做轉換)

給定兩字串 word1、word2，如果 word1 與 word2 是密切的(close)則回傳 true，否則回傳 false

範例2:

限制:

  • 1 <= word1.length, word2.length <= 105

  • 字串 word1 及 word2 都只有小寫英文字母

解題:

條件1相對單純，這題主要判斷如何讓 操作2 的條件成立，其實只要去判斷兩字串中，字母分布率是否相同即可

也就是說，假如兩字串是 cabbba、abbccc，你會發現兩字串字母分布率相同，分別是 {“c”: 1, “b”: 3, “a”: 2} 及 {“c”: 3, “b”: 2, “a”: 1} (都是 1、2、3)；也可以加上一些判斷邏輯略過根本不需要計算分母分布率的情況

雙子星

應徵職位 : 雲端開發工程師

雙子星透過 104 邀請面試，公司位於新竹京元電子附近，從高鐵站搭小黃前往大約 20 分鐘就能抵達，公司是工研院出來的團隊，是一間蠻新的公司，主要是在做雲端服務相關的產品，透過 openstack 做應用，目前著重在 PaaS 及 SaaS 的開發。

這次的面試過程分成三個階段
第一階段先跟技術協理 Kenneth 做自我介紹，主管也介紹公司目前有三個部門正在找人，分成了   

  • Infra Team: PaaS的開發
  • Software Team: SaaS的開發
  • CAT Team: 跟客戶談需求並解決客戶問題的一個 Team，所以公司內的技術及產品基本上都會碰 (所以說要不排斥面對客戶)

公司會透過你的能力來決定工作崗位，也採取自由的制度轉換工作崗位，而就我面試的狀況，公司也願意讓我自己選擇想要的方向。因為是應屆畢業生，因此公司會建議我從 CAT Team 開始做起，未來也可以根據自己的所好來決定未來的方向。

第二階段就由這三個部門的組長跟我聊 (1 vs. 3 的概念)，主要問了有關過去的經驗，大學的專題、研究所的論文，並沒有問我太多有關雲端的問題，接著就讓我自由提問。

第三階段，主要就跟 HR 經理 Mindy 閒話家常，因為對於薪水的部分在第一階段 Kenneth 就已經有先詢問了，此外也剛好談到北漂的議題，這一階段短短只有10分鐘，也是讓我蠻訝異的，如果有進一步的消息會再通知。

11/29 二面。

這次的面試只是跟 CAT Team Lead - Zero 進一步的對談，在一面所見到的是 CAT Team 底下的一員，主要是再進一步的告訴我 CAT Team 這一個團隊主要的工作範疇。

結果: Offer get

趨勢科技

應徵職位 : Python Developer

目前在網路上找不到這一個職缺，主要是 104獵頭 主動透過信件向我詢問的

相信讀資訊領域的應該都聽過趨勢科技，而難得有這一個機會，工作內容也跟我找的很相似，當然是二話不說地直接答應囉!

在台北找工作最大的好處是，基本上面試的公司，捷運都能到，相當的便利，而這一次的面試主要分成兩個階段。

第一階段，如同我上面所說的，104獵頭主動詢問我意見，所以一開始獵頭(後面簡稱 Hunter) 告知我趨勢有這一個工作職缺，問我有沒有這一個意願前往面試，在談話的過程中，問到了我許多有關程式開發的想法，這個就根據每個人經驗來回答囉，而 Hunter 也告知我， 要我先到 Codility 進行測試，完成測試之後會再進一步幫我通知該部門，並等待面試通知。

Codility 測驗內容不難，當時我所拿到的題目，主要透過 list 及 一點邏輯就能完成，主要考量的是，Codility 還會評估你的程式碼寫得有沒有效率! 也就是說，測資是有設計過的，所以不僅測驗你程式碼的準確率，還有時間複雜度。

凡事都是 計畫敢不上變化的，在原本規劃面試的行程中， 收到了 趨勢科技 下一個階段的面試邀約，來得太突然，導致原本當天面試完畢要回高雄的我，就直接轉戰台北了。

公司在大安區，台北捷運真的很方便! 

第二階段是跟趨勢科技的 技術經理 Jo面談，主要告訴我這一個職缺在做什麼，不如以往大家所了解的趨勢科技就是做資安寫防毒的，這部門不做資安，不搞大數據，做的是產品的 license ，因為趨勢最近收購了一間美國的公司 tipping point，除了要維持原本美國那裏客戶的服務，同時間也要將此產品做新的開發。

而相對的也會透過 Python 的框架 Django 進行網頁的開發，除此之外還要處理客戶所提出的 bug，進行修復的工作。

a = [1, 2], b = [3, 4]
a.append(b)

a.extend(b)

除此之外還問到了一些

  • Encrypt vs. Hash

Jo 也在這一次的面談中，也詢問了很多團隊合作的問題，並強調溝通的重要性，因為這一個部門在 2017.11 剛成立，所以該部門的運作相當的迅速，除此之外也要不排斥用英文溝通，因為還有很多問題是從美國的客戶來的，要根據相對應的問題而做出解決方案。

Git 細節

Posted on 2018-11-24 Edited on 2021-11-23 In 未分類

這篇文章內容主要探討 Git 相似的名詞，記錄下來避免未來忘記

  • master 及 origin/master 
  • fetch 及 pull
  • rebase 及 merge

master vs. origin/master

master: 本地端分支
origin/master: 遠端分支

更白話的說，本地端有一個 master 分支，當我們進行 git push，會複製一份本地端 master 到遠端 origin ，也就是 origin/master 分支囉!

fetch vs. pull

fetch: 從 remote repository 取得最新的版本，這裡舉動同時會自動建立分支FETCH_HEAD，而這被撈下來的版本會自動放進此分支
pull: 從 remote repository 取得最新的版本並直接將遠端分支 origin/master 進行合併提交

一個 pull 等同於 
1. fetch 
2. merge origin/master

rebase vs. merge

假設，目前我們有兩個分支 A、B，且此時我們在 分支 B

rebase: 將 A 當作基底，並將 B分支 的內容嫁接在 分支A 後方 (兩條變一條的概念)，值得注意的一點是，因為 提交位置 更動了，所以 SHA-1 會重新計算!
merge: 將 A 合併進分支 B，並產生一個額外的 提交

內文

URL (Uniform Resource Locator): 俗稱 網址，詳細來說就是在互聯網上你瀏覽各個網頁的位址

e.g. http://example.com:8000/foo?p=foo#test

  • http: 發出請求所使用的協議 (Protocol)
  • example.com/foo: 請求對象之Host
  • 8000: 使用哪一個端口 (Port)
  • ?p=foo: ?後方代表 額外請求的資訊 (name = value)
  • #test: 使用標記 (fragment) 前往相同網頁下指定的內容

網址最後要不要加 slash?

https://webmasters.googleblog.com/2010/04/to-slash-or-not-to-slash.html

而使用者輸入完網址按下 enter 發送的 **請求(request)**，或者是伺服器的 回覆(response)，一個 HTTP Message，俗稱 訊息， 就會產生，這兩者唯一的差別就在於 Start-line，對於請求來說是發送 request-line，而 回覆 則是 status-line

Start-line = request-line / status-line

一個 HTTP Message包含下列內容

  • Request-Line: method SP request-target SP HTTP-Version CRLF (SP: 空白)
  • Header Field CRLF
  • CRLF: carriage return + line feed，也就是俗稱的 \r\n
  • Message-body

所以一個完整的 HTTP Message 就會長這樣

Request = Start-line
                  *(Header Field) CRLF
                  CRLF
                  Message-body
